Core Data Engineering Concepts

Deep dive into essential concepts every data engineer must master

The Data Engineering Lifecycle

The data engineering lifecycle represents the end-to-end journey of data from generation to consumption. Understanding this lifecycle is fundamental to designing robust data systems.

1. Generation

Data originates from various sources:

  • Application databases - transactional systems (OLTP)
  • APIs - REST, GraphQL, webhooks
  • Event streams - user clicks, IoT sensors
  • File systems - CSV, JSON, logs
  • Third-party data - external vendors, partners

2. Storage

Raw and processed data storage:

  • Data Lakes - S3, Azure Data Lake (raw, schema-on-read)
  • Data Warehouses - Redshift, Snowflake (structured, schema-on-write)
  • Lakehouses - Delta Lake, Iceberg (best of both)
  • Object Storage - S3, GCS, Azure Blob

3. Ingestion

Moving data from source to storage:

  • Batch - scheduled jobs, full/incremental loads
  • Streaming - real-time processing, Kinesis, Kafka
  • CDC - capturing database changes incrementally
  • Tools - AWS Glue, Fivetran, Airbyte

4. Transformation

Processing and enriching data:

  • ETL - Extract, Transform, Load (traditional)
  • ELT - Extract, Load, Transform (modern approach)
  • Tools - dbt, Spark, AWS Glue, Dataflow
  • Operations - cleaning, joining, aggregating, enriching

5. Serving

Making data available for consumption:

  • Analytics - BI dashboards, ad-hoc queries
  • ML Models - feature stores, model training
  • APIs - data products, microservices
  • Reports - scheduled exports, email alerts

6. Governance

Ensuring quality, security, and compliance:

  • Data Quality - validation, profiling, monitoring
  • Security - encryption, access control, auditing
  • Privacy - GDPR, CCPA, PII masking
  • Lineage - tracking data origins and transformations

Data Architecture Principles

Scalability

Horizontal vs. Vertical Scaling:

  • Horizontal - add more nodes (distributed systems, cloud-native)
  • Vertical - increase resources per node (limited, expensive)

Design Patterns:

  • Partitioning and sharding data
  • Load balancing across nodes
  • Auto-scaling based on demand
  • Caching frequently accessed data

Loose Coupling

Decoupling Components:

  • Microservices architecture
  • Message queues (SQS, Kafka) between services
  • API-first design
  • Event-driven architectures

Benefits:

  • Independent scaling of components
  • Easier testing and debugging
  • Technology flexibility per service
  • Reduced blast radius of failures

Reversible Decisions

Avoid Lock-in:

  • Use open formats (Parquet, ORC, Avro)
  • Abstract vendor-specific APIs
  • Infrastructure as Code (Terraform, CloudFormation)
  • Multi-cloud strategies where appropriate

Version Control Everything:

  • Schema evolution and versioning
  • Code in Git repositories
  • Infrastructure state management
  • Rollback mechanisms

Batch vs Streaming Processing

Batch Processing

Characteristics:

  • Processes large volumes of data at scheduled intervals
  • High throughput, high latency (minutes to hours)
  • Simpler to implement and debug
  • Lower cost per record processed

Use Cases:

  • Daily sales reports and analytics
  • Monthly aggregations for billing
  • Historical data analysis
  • ML model training on historical data

Tools:

  • Apache Spark (batch mode)
  • AWS Glue ETL jobs
  • dbt transformations
  • Cron-scheduled scripts

Streaming Processing

Characteristics:

  • Processes data in real-time as it arrives
  • Low latency (milliseconds to seconds)
  • More complex to implement (state management)
  • Higher infrastructure cost

Use Cases:

  • Fraud detection (must be real-time)
  • Real-time dashboards and monitoring
  • Event-driven architectures
  • Live recommendation engines

Tools:

  • Apache Kafka + Kafka Streams
  • Amazon Kinesis (Data Streams, Analytics)
  • Apache Flink, Spark Streaming
  • AWS Lambda (event-driven)

When to Choose What?

  • Batch if: latency requirements are >1 hour, cost is critical, simple aggregations
  • Streaming if: latency must be <1 minute, need event-driven triggers, real-time decisions
  • Lambda Architecture - combine both: batch for accuracy, streaming for speed
  • Kappa Architecture - streaming-only with reprocessing capabilities

Change Data Capture (CDC)

CDC captures incremental changes in source databases and streams them to downstream systems in near real-time. This is critical for keeping data warehouses in sync without full table scans.

1. Log-Based CDC (Best Practice)

Reads database transaction logs (binlog, WAL):

  • Tools: Debezium, AWS DMS, Fivetran
  • Pros: No schema changes, all operations captured, low overhead
  • Cons: Requires log access, complex setup
  • Example: MySQL binlog → Debezium → Kafka → S3/Redshift

2. Trigger-Based CDC

Database triggers write changes to a shadow table:

  • Pros: Simple to implement, works on any DB
  • Cons: Performance overhead, schema changes required
  • Use case: Legacy systems without log access

3. Timestamp-Based CDC

Queries tables with updated_at columns:

  • Pros: Very simple, no special permissions
  • Cons: Misses deletes, clock skew issues
  • Use case: Small tables, append-only data

CDC in AWS

  • AWS DMS - Full load + CDC from RDS/on-prem to S3/Redshift/Kinesis
  • Debezium on EC2/ECS - Open-source, log-based CDC to Kafka/Kinesis
  • Pattern: RDS (MySQL/Postgres) → DMS → Kinesis → Lambda → S3 → Glue → Redshift

Modern Storage Layers

Object Storage (S3)

Why S3 Dominates Data Engineering:

  • Infinitely scalable, pay-per-use
  • 99.999999999% durability (11 9s)
  • Lifecycle policies (auto-archive to Glacier)
  • Integrates with all AWS data services
  • Supports all file formats

Best Practices:

  • Organize with prefixes: s3://bucket/year=2024/month=01/day=15/
  • Enable versioning for critical data
  • Use S3 Select for querying without downloading
  • Partition data for query performance

Lakehouse Architecture

Combines Data Lake + Data Warehouse:

  • Store raw data in S3 (like a lake)
  • ACID transactions via Delta Lake/Iceberg (like a warehouse)
  • Schema evolution and time travel
  • Single source of truth for all workloads

Technologies:

  • Delta Lake - Databricks, open-source
  • Apache Iceberg - Netflix, AWS support
  • Apache Hudi - Uber, incremental processing

AWS Implementation:

  • S3 + Glue Catalog + Athena + Iceberg tables
  • EMR with Delta Lake

Parquet Format

Why Parquet is the Standard:

  • Columnar storage - read only needed columns (vs row-based CSV)
  • Compression - 10x smaller than JSON/CSV (Snappy, Gzip)
  • Predicate pushdown - skip irrelevant row groups
  • Schema embedded - self-describing files
  • Widely supported - Spark, Athena, Redshift Spectrum

Performance Example:

  • 1TB CSV file with 100 columns
  • Query needs only 5 columns
  • CSV: reads entire 1TB
  • Parquet: reads ~50GB (10x faster, 95% cost savings)

When to Use:

  • Analytical queries (OLAP)
  • Data lake storage
  • Large datasets (>1GB)

ETL vs ELT

ETL (Traditional)

Extract → Transform → Load

Transform data before loading into warehouse:

  • Extract from source systems
  • Transform in dedicated ETL server (Informatica, Talend, AWS Glue)
  • Load cleaned data into warehouse

Pros:

  • Smaller warehouse storage (only clean data)
  • Better for limited compute in warehouse
  • Compliance: sensitive data never hits warehouse

Cons:

  • Slower (transformation bottleneck)
  • Harder to debug (data already transformed)
  • Inflexible (schema changes require ETL re-development)

When to Use:

  • On-premise warehouses with limited compute
  • PII/PHI data requiring masking before storage

ELT (Modern Approach)

Extract → Load → Transform

Load raw data first, transform inside warehouse:

  • Extract from source systems
  • Load raw data into warehouse (S3, Redshift, Snowflake)
  • Transform using SQL (dbt, stored procedures)

Pros:

  • Faster ingestion (no transformation delay)
  • Leverage warehouse compute (Redshift scales easily)
  • Flexibility: re-transform raw data anytime
  • Easier debugging (raw data preserved)

Cons:

  • More storage needed (raw + transformed)
  • Raw data may contain sensitive info

When to Use:

  • Cloud warehouses with elastic compute (Redshift, Snowflake)
  • Need to re-process historical data
  • This is the default for modern data stacks

Modern ELT Stack (AWS)

  • Extract + Load: Fivetran, Airbyte, AWS Glue, DMS → S3
  • Transform: dbt (SQL-based transformations), Athena, Redshift
  • Orchestrate: Airflow, Dagster, Step Functions

Data Modeling

Star Schema (Most Common)

Structure:

  • Fact Table - center, contains metrics (sales_amount, quantity)
  • Dimension Tables - surround fact, contain attributes (customer, product, date)
  • Foreign keys in fact table → primary keys in dimensions

Example: Sales Analytics

Fact: fact_sales (sale_id, date_id, customer_id, product_id, store_id, sales_amount, quantity)
Dimensions:
  - dim_date (date_id, date, month, quarter, year)
  - dim_customer (customer_id, name, segment, region)
  - dim_product (product_id, name, category, brand)
  - dim_store (store_id, name, city, state)

Pros:

  • Simple queries (one JOIN per dimension)
  • Fast aggregations
  • Easy for BI tools to understand

Cons:

  • Denormalized dimensions (data duplication)
  • Harder to maintain dimensional hierarchies

Snowflake Schema

Normalized version of star schema - dimensions are normalized into sub-dimensions:

  • dim_product → dim_category → dim_brand (3 tables instead of 1)
  • Reduces data duplication
  • More JOINs required (slower queries)

When to Use:

  • Storage is expensive (on-premise warehouses)
  • Need strict normalization
  • Most cloud warehouses prefer star schema

Normalization vs Denormalization

Normalization (OLTP - transactional systems):

  • Reduce data duplication
  • Third Normal Form (3NF) - minimize redundancy
  • Good for writes, slower for reads
  • Use for: production databases (RDS, Postgres)

Denormalization (OLAP - analytical systems):

  • Combine tables to reduce JOINs
  • Faster reads, slower writes
  • More storage (duplicate data)
  • Use for: data warehouses (Redshift, BigQuery)

Query Optimization

1. Partitioning

Split large tables into smaller, manageable chunks:

  • S3: s3://bucket/year=2024/month=01/day=15/data.parquet
  • Athena/Glue: PARTITIONED BY (year, month, day)
  • Redshift: Distribution + Sort keys
  • Benefit: Query only relevant partitions (10x-100x speedup)

Example:

-- Instead of scanning 365 days:
SELECT * FROM sales WHERE date = '2024-01-15'

-- With partitioning, only scan 1 partition:
SELECT * FROM sales WHERE year=2024 AND month=1 AND day=15

2. Indexing

Speed up lookups on specific columns:

  • B-tree indexes: equality/range queries
  • Hash indexes: exact matches only
  • Redshift Sort Keys: define sort order
  • Redshift Dist Keys: minimize data shuffling

When to Index:

  • Columns in WHERE clauses
  • JOIN keys
  • ORDER BY / GROUP BY columns

3. Predicate Pushdown

Apply filters early in query execution:

  • Parquet: skip entire row groups based on min/max stats
  • Athena: only read matching S3 files
  • Spark: push filters to storage layer

Example:

-- Bad: Filter after JOIN
SELECT * FROM large_table t1
JOIN huge_table t2 ON t1.id = t2.id
WHERE t1.date = '2024-01-15'

-- Good: Filter before JOIN
SELECT * FROM large_table t1
WHERE t1.date = '2024-01-15'
JOIN huge_table t2 ON t1.id = t2.id

4. Column Pruning

Select only needed columns (especially with Parquet):

-- Bad: Read all 100 columns
SELECT * FROM wide_table

-- Good: Read only 3 columns
SELECT customer_id, sales_amount, date FROM wide_table

-- With Parquet: 97% I/O reduction!

5. Materialized Views

Pre-compute expensive aggregations:

  • Redshift: CREATE MATERIALIZED VIEW daily_sales AS SELECT date, SUM(amount) FROM sales GROUP BY date
  • Refresh periodically (incremental or full)
  • Trade-off: storage vs query speed

6. Compression

Reduce I/O and storage costs:

  • Parquet: Snappy (fast), Gzip (smaller)
  • Redshift: Auto-compression encoding per column
  • Benefit: 5-10x storage savings, faster scans

Always Use EXPLAIN

  • EXPLAIN shows query execution plan
  • Look for: full table scans, expensive JOINs, missing indexes
  • Redshift: EXPLAIN SELECT ...
  • Athena: Query stats in console

Orchestration (Airflow, Dagster, etc.)

Orchestration tools schedule, monitor, and manage complex data pipelines with dependencies.

Apache Airflow (Industry Standard)

Concepts:

  • DAG - Directed Acyclic Graph (pipeline definition)
  • Tasks - individual units of work (Python functions, bash commands)
  • Operators - task templates (PythonOperator, BashOperator, S3Operator)
  • Scheduler - triggers DAGs on schedule or events
  • Executor - runs tasks (Local, Celery, Kubernetes)

AWS Deployment:

  • MWAA - Managed Workflows for Apache Airflow (fully managed)
  • Self-hosted - EC2 + RDS for metadata DB

Pros:

  • Massive community, 1000+ integrations
  • Python-based (flexible)
  • Great UI for monitoring

Cons:

  • Complex setup (unless using MWAA)
  • DAGs defined in code (not GUI)

Dagster (Modern Alternative)

Key Differences from Airflow:

  • Software-defined assets - focus on data assets, not tasks
  • Type safety - Python type hints for data validation
  • Testing-first - easier to unit test pipelines
  • Data lineage - built-in asset dependency tracking

When to Use:

  • New projects (greenfield)
  • Data quality is critical
  • Need strong testing/observability

AWS Step Functions

Serverless orchestration for AWS services:

  • Define workflows in JSON (State Language)
  • Native integrations: Lambda, Glue, EMR, Batch, ECS
  • Visual workflow editor
  • Pay-per-execution (no servers to manage)

Use Case Example:

1. Lambda triggers on S3 upload
2. Step Function starts:
   a. Glue Crawler updates schema
   b. Glue ETL job processes data
   c. Lambda validates output
   d. SNS sends success notification

Pros:

  • Fully managed, serverless
  • Great for AWS-only pipelines

Cons:

  • AWS lock-in
  • Limited to AWS services

Other Tools

  • Prefect - Python-based, easier than Airflow
  • Luigi - Spotify's tool (simpler than Airflow)
  • AWS Glue Workflows - basic orchestration for Glue jobs
  • dbt Cloud - orchestrates dbt transformations

Orchestration Best Practices

  • Idempotency - tasks should produce same result if re-run
  • Incremental processing - process only new data, not full reloads
  • Retry logic - auto-retry transient failures (network, API limits)
  • Alerting - SNS/email on failures
  • Backfills - ability to reprocess historical data

DataOps & FinOps

DataOps (Data + DevOps)

Apply DevOps principles to data pipelines:

  • Version Control - Git for all code, schemas, configs
  • CI/CD - automated testing + deployment
    • Unit tests for transformations (dbt tests)
    • Integration tests on sample data
    • GitHub Actions / GitLab CI
  • Monitoring & Observability
    • Data quality checks (Great Expectations, dbt tests)
    • Pipeline health dashboards (Airflow UI, DataDog)
    • Schema drift detection
  • Data Lineage - track data from source to dashboard
    • Tools: AWS Glue Data Catalog, Amundsen, DataHub
  • Environment Management
    • Dev / Staging / Prod separation
    • Infrastructure as Code (Terraform)

DataOps Workflow Example:

1. Developer writes dbt model
2. Git commit triggers CI pipeline:
   - Run unit tests (dbt test)
   - Deploy to dev environment
   - Run integration tests
3. PR review + approval
4. Merge to main → auto-deploy to prod
5. Monitoring alerts on data quality issues

FinOps (Financial Operations)

Optimize cloud costs without sacrificing performance:

  • Cost Visibility
    • AWS Cost Explorer - track spend by service
    • Tag resources (project, team, env)
    • Athena query costs (data scanned)
  • Right-Sizing
    • Redshift: choose instance types based on workload
    • EMR: spot instances for batch jobs (70% savings)
    • Lambda: optimize memory allocation
  • Storage Optimization
    • S3 Lifecycle policies: archive to Glacier after 90 days
    • Delete old logs, temp files
    • Use Parquet instead of JSON (10x smaller)
    • Enable S3 Intelligent-Tiering
  • Query Optimization = Cost Savings
    • Athena charges per TB scanned
    • Partition data: WHERE year=2024 scans 1/5th of data (80% savings)
    • Use Parquet + column pruning
  • Auto-Scaling & Scheduling
    • Pause Redshift clusters during off-hours
    • EMR auto-scaling for variable workloads
    • Use Lambda instead of 24/7 EC2 for light workloads
  • Reserved Instances & Savings Plans
    • 1-3 year commitments for 30-70% discounts
    • Use for predictable workloads (Redshift, RDS)

Cost Monitoring Example:

Daily automated report:
- S3 storage growth: +50GB (check for temp files)
- Athena scans: 2TB today (optimize with partitions?)
- Redshift idle time: 6 hours (pause overnight?)
- Top expensive queries (EXPLAIN + optimize)

DataOps + FinOps Best Practices

  • Shift-left cost awareness - developers think about cost during design
  • Automate everything - no manual deployments or checks
  • Monitor continuously - catch issues before they become expensive
  • Document runbooks - how to respond to alerts
  • Blameless postmortems - learn from failures
